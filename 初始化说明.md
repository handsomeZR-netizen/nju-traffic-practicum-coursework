# 智慧交通项目初始化说明

## 1. 目的

这份文档用于说明本项目在本地和云端的最小初始化要求、目录职责、不可改动项以及启动前检查步骤。

当前项目的核心目标是：

- 保证腾讯开悟训练/评测环境可以正常启动
- 保证算法代码可单独迭代
- 避免因为误改框架配置目录导致训练脚本直接失败

---

## 2. 目录职责

项目根目录下当前主要目录含义如下：

- `agent_ppo/`
  主要提交算法目录，当前实际训练和评测主要依赖这里

- `agent_diy/`
  当前主提分目录，平台 `DIY` 槽位加载的就是这里

- `conf/`
  项目级配置目录

- `conf/kaiwudrl/`
  开悟框架运行配置目录，包含 `learner.toml`、`configure.toml`、`aisrv.toml` 等运行时关键文件

- `tools/`
  开悟框架脚本目录，包含 `change_sample_server.sh`、`modelpool_start.sh`、`stop.sh` 等启动依赖脚本

- `train_test.py`
  本地/云端快速训练检查入口

- `kaiwu.json`
  项目基础元信息

---

## 3. P0 不可改动项

以下内容属于 **P0 级重要性**，默认视为框架依赖，不允许随意修改、删除、重命名或精简。

### 3.1 `conf/kaiwudrl/`

该目录是训练启动的硬依赖。至少应包含以下文件：

- `actor.toml`
- `aisrv.toml`
- `client.toml`
- `configure.toml`
- `learner.toml`
- `monitor.yaml`

如果该目录为空、少文件、文件名变化，`train_test.py` 以及 `tools/change_sample_server.sh` 会直接失败。

### 3.2 `tools/`

该目录是训练控制脚本集合，至少本项目会直接依赖这些脚本：

- `tools/change_sample_server.sh`
- `tools/modelpool_start.sh`
- `tools/stop.sh`

此外 `tools/` 下还包含大量框架辅助脚本，不建议删减，也不建议为了“精简上传”而手动裁剪该目录。

### 3.3 原则

- `conf/kaiwudrl/` 和 `tools/` 视为框架保留区
- 不在这些目录中做业务逻辑开发
- 不对这些目录做“合并文件”“删文件”“改名字”“挪位置”
- 上传到云端时必须完整保留

---

## 4. 允许改动项

当前允许且建议主要改动的区域如下：

- `agent_ppo/agent.py`
- `agent_ppo/feature/definition.py`
- `agent_ppo/model/model.py`
- `agent_ppo/conf/conf.py`
- `agent_ppo/workflow/train_workflow.py`
- 根目录文档文件

如无明确必要，尽量不要修改：

- `conf/` 根目录下已有的 app/algo 配置
- `train_test.py` 以外的框架入口

---

## 5. 云端初始化要求

### 5.1 云端项目必须具备的关键结构

```text
intelligent_traffic_lights/
├─ agent_ppo/
├─ agent_diy/
├─ conf/
│  ├─ algo_conf_intelligent_traffic_lights.toml
│  ├─ app_conf_intelligent_traffic_lights.toml
│  ├─ configure_app.toml
│  └─ kaiwudrl/
│     ├─ actor.toml
│     ├─ aisrv.toml
│     ├─ client.toml
│     ├─ configure.toml
│     ├─ learner.toml
│     └─ monitor.yaml
├─ tools/
├─ kaiwu.json
└─ train_test.py
```

### 5.2 常见问题

最常见的启动失败不是算法错误，而是：

- `conf/kaiwudrl/` 目录存在但里面是空的
- `tools/` 被平台裁剪或没有完整上传
- 云端只传了 `agent_ppo/`，没有传框架依赖目录

---

## 6. 启动前自检命令

在云端终端执行：

```bash
cd /data/projects/intelligent_traffic_lights
pwd
ls -lah conf
ls -lah conf/kaiwudrl
ls -lah tools
```

如果 `conf/kaiwudrl` 为空，说明项目包不完整，先补齐文件再启动训练。

进一步检查关键文件：

```bash
cd /data/projects/intelligent_traffic_lights
stat conf/kaiwudrl/learner.toml
stat conf/kaiwudrl/configure.toml
stat tools/change_sample_server.sh
stat tools/modelpool_start.sh
stat tools/stop.sh
```

---

## 7. 标准启动方式

推荐在项目根目录执行：

```bash
cd /data/projects/intelligent_traffic_lights
/bin/python3 train_test.py
```

不建议在其他目录下直接执行相对路径脚本。

---

## 8. 当前项目开发约定

为了减少后续反复踩坑，当前项目采用以下约定：

- 算法开发优先集中在 `agent_ppo/`
- 冠军思路迁移后的实际提交版本集中在 `agent_diy/`
- `conf/kaiwudrl/` 和 `tools/` 视为 P0，不参与代码精简
- 上传到云端时，优先保证目录完整，再考虑代码优化
- 如果云端只能复制粘贴文件内容，优先恢复 `conf/kaiwudrl/`，不要先改算法
- 若要上传当前主方案，优先使用 `build_upload_diy_package.py` 生成的 `upload_diy_minimal/`

---

## 9. 出错时的处理顺序

如果训练失败，按以下顺序排查：

1. 先看 `conf/kaiwudrl/` 是否为空
2. 再看 `tools/` 关键脚本是否存在
3. 再看 `train_test.py` 是否从项目根目录启动
4. 最后再排查算法代码本身

不要一开始就怀疑 PPO 或模型逻辑，因为多数启动失败都发生在训练框架初始化阶段。

---

## 10. 结论

本项目中：

- `conf/kaiwudrl/` 是 **P0 级目录**
- `tools/` 是 **P0 级目录**
- 这两个目录不能改、不能删、不能合并、不能精简

后续如果需要继续整理项目结构，应只整理算法目录和文档，不碰框架目录。
